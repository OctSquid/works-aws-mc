# 仕様

- AWSを使用し，Minecraft Paper 26.1 + GayserMC + Floodgateサーバーを構築する．
- つまり，Java版とBedrock版の両方に対応する．
- すべてIaCで構築する．
- Terraformを使用して，AWSのリソースを構築する．
- コードベースはGitで管理する
- できるだけ低コストで運用できる仕組みにする．
- GitHub Actionsを使用して，AMIのビルドとIaCのコードを自動デプロイする．
- AMIを使用して，Minecraftサーバーの構築・起動・停止を自動化する．
- Discord Botを使用して，必要な時のみサーバーを構成・起動する．
- スポットインスタンスを使用して，コストを削減する．(m6g.large = Graviton/arm64 を使用する．server.json の ec2 で変更可能)
- AWS Lambdaを使用して，DiscordのSlash Commandsからのリクエストに応じてサーバーを構成・起動する．
- サーバーが停止されたとき，インスタンスを削除する．
- Cron + EventBridge + Rcon を使用して，15分間プレイヤーがいない場合にサーバーを自動停止する．
- サーバーの自動停止，スポットインスタンスの中断通知はDiscordに通知する．
- また，スポットインスタンスの中断が通知された場合．サーバー内にRconで通知する．
- Route53を使用して，独自ドメインでサーバーにアクセスできるようにする．(Estatic IP使用せずにコストを削減する．つまり，EC2インスタンスが起動した時にIPアドレスをRoute53のドメインに紐づける．)
- プラグイン・プラグインのバージョンは `plugins.json` に記述する．
- プラグインはビルド時にCurseForge,SpigotMC,GitHubなどからダウンロードする．
- プラグインのバージョンアップは，`plugins.json` の更新とGitHub Actionsの実行で自動化する．
- サーバーの設定ファイル群やプラグインの設定ファイル群は，Git上で管理する．
- サーバーの設定ファイル群やプラグインの設定ファイル群は，GitHub Actionsを使用して，AMIのビルド時に自動で反映されるようにする．
- EBSのスナップショットを使用して，サーバーのデータ(プレイヤーデータ・ワールドデータ・プラグインのプレイヤーデータ)をバックアップする．
- ローカルでテスト環境を構築できるようにする．
- Lambdaの言語などはTypeScriptを使用する．
